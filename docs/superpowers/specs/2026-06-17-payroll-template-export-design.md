# 劳务工资代发模板生成系统设计

## 1. 目标

建设一个劳务公司使用的工资、奖金、报销、劳务费等代发模板生成系统。第一版只负责基础资料维护、工资批次制作、银行 Excel 模板导出、数据权限隔离和操作日志审计，不做银行提交、回盘、发放状态跟踪和审批流。

系统核心体验是：先批量选择代发人员，人员库没有的再手工逐行录入；填写金额、摘要、备注后自动计算总人数和总发金额；导出时选择付款单位和银行模板，生成对应银行要求的 Excel 文件。

## 2. 用户角色与权限

### 2.1 普通录入人员

- 只能查看、编辑、删除自己创建的代发人员、工资计发单位、工资批次和导出记录。
- 可以制作工资批次、修改批次明细、导出银行模板。
- 在批次中录入的新人员、新付款单位会自动保存到自己的资料库。

### 2.2 管理员

- 可以查看全部人员、单位、批次、导出记录和操作日志。
- 可以调整所有用户创建的数据。
- 管理员调整他人数据时必须记录操作日志。

## 3. 功能模块

### 3.1 基础信息

#### 代发人员库

字段：

- 姓名
- 身份证号，必填，用于去重
- 手机号码
- 银行账号
- 户名
- 行名
- 银行类型
- 银行类别
- 联行行号
- 创建人
- 创建时间
- 更新时间

规则：

- 新增人员时身份证号必填。
- 保存批次时，如果明细中的身份证号不存在，则自动新增人员。
- 如果身份证号已存在，则复用该人员，并可更新手机号、银行账号、户名、行名、银行类型、银行类别、联行行号等可变信息。
- 普通用户只能看到自己创建的人员，管理员可以看到全部。

#### 工资计发单位库

字段：

- 银行账号
- 户名
- 行名
- 银行类型
- 银行类别
- 联行行号
- 创建人
- 创建时间
- 更新时间

规则：

- 银行账号用于去重。
- 导出时优先选择已有付款单位。
- 如果付款单位不存在，可在导出或维护页面直接录入，保存后自动进入单位库。
- 如果银行账号已存在，则复用该单位，并可更新户名、行名、银行类型、银行类别、联行行号。
- 普通用户只能看到自己创建的单位，管理员可以看到全部。

### 3.2 工资批次制作

批次只保存本次要发给哪些人、金额、摘要和备注，不绑定付款账号，也不固定导出银行。

批次字段：

- 批次名称
- 发放日期
- 默认摘要
- 备注
- 总人数
- 总发金额
- 创建人
- 创建时间
- 更新时间

批次明细字段：

- 批次 ID
- 人员 ID，可为空，手工新录入人员保存后回填
- 姓名
- 身份证号
- 手机号
- 银行账号
- 户名
- 行名
- 银行类型
- 银行类别
- 联行行号
- 金额
- 摘要
- 备注
- 行号

页面流程：

1. 新建工资批次。
2. 填写批次名称、发放日期、默认摘要、备注。
3. 点击批量选择人员，从人员库多选加入明细。
4. 人员库没有的，在明细表中新增空行并手工录入。
5. 填写金额、摘要、备注。
6. 系统自动计算总人数和总发金额。
7. 保存批次。
8. 后续允许继续修改批次和明细，每次修改记录操作日志。

录入体验要求：

- 人员选择弹窗支持按姓名、身份证号、手机号、银行卡号搜索。
- 人员选择支持多选，一次加入工资明细表。
- 已加入当前批次的人员不重复添加。
- 明细表支持新增空行、删除行、批量设置摘要。
- 金额输入后实时刷新总人数、总发金额。
- 页面显示缺银行账号人数、缺联行行号人数，便于导出前检查。
- 摘要建议提供工资、奖金、报销、劳务费等快捷选项。

### 3.3 银行模板导出

导出时选择：

- 工资批次
- 工资计发单位
- 银行模板：建设银行、中国银行

导出规则：

- 同一工资批次可以导出多次。
- 导出不修改批次内容，只生成导出文件和导出记录。
- 不同银行模板可选择不同付款账号。
- 批次导出后仍允许修改，再次导出生成新的导出记录，不覆盖历史记录。
- 收款人允许混合不同银行，系统按所选银行模板统一导出。

导出前校验：

- 身份证号不能为空。
- 银行账号不能为空。
- 户名不能为空。
- 金额必须大于 0。
- 他行收款时，若银行模板要求联行号或 CNAPS 号，缺失时给出提示。

#### 建设银行模板映射

当前模板文件：`建设银行工资模版.xlsx`

| 模板字段 | 系统字段 | 规则 |
| --- | --- | --- |
| 序号（必填） | 自动生成 | 从 1 开始按明细顺序生成 |
| 账号（必填） | 人员银行账号 | 必填 |
| 户名（必填） | 人员户名 | 必填 |
| 金额（必填） | 明细金额 | 必须大于 0 |
| 跨行标识 | 人员银行类型/银行类别 | 建行填 0，他行填 1 |
| 行名 | 人员行名 | 跨行业务与联行行号不能同时为空 |
| 联行行号 | 人员联行行号 | 跨行业务与行名不能同时为空 |
| 摘要 | 明细摘要 | 显示在收款账户流水明细中 |
| 备注 | 明细备注 | 可为空 |

#### 中国银行模板映射

当前模板文件：`中国银行工资模版.xls`

| 模板字段 | 系统字段 | 规则 |
| --- | --- | --- |
| 卡号/账号 | 人员银行账号 | 必填 |
| 户名 | 人员户名 | 必填 |
| 金额 | 明细金额 | 必须大于 0 |
| 收款行类型 | 人员银行类型/银行类别 | 中行填中行，他行填他行 |
| 收款行CNAPS号 | 人员联行行号 | 他行收款时必填 |
| 证件类型 | 固定值 | 默认居民身份证 |
| 证件号码 | 人员身份证号 | 必填 |
| 用途 | 明细摘要 | 建议限制为工资、奖金 |
| 附言 | 明细备注 | 可为空 |

### 3.4 导出记录

字段：

- 工资批次 ID
- 付款单位 ID
- 银行模板类型
- 文件名
- 文件路径
- 总人数
- 总金额
- 导出人
- 导出时间

规则：

- 每次导出都新增一条记录。
- 历史导出记录不覆盖。
- 普通用户只看自己的导出记录，管理员看全部。

### 3.5 操作日志

记录动作：

- 新增、修改、删除人员
- 新增、修改、删除工资计发单位
- 新建、修改、删除工资批次
- 新增、修改、删除批次明细
- 导出银行模板
- 管理员调整他人数据

日志字段：

- 操作人
- 操作时间
- 操作类型
- 业务模块
- 业务对象 ID
- 修改前内容
- 修改后内容
- IP 地址
- 设备或浏览器信息

## 4. 技术选型

### 4.1 前端

推荐：`Vue 3 + Element Plus`

原因：

- 适合后台管理系统。
- 表格、弹窗、表单、日期选择、菜单、分页、权限菜单组件成熟。
- 工资批次制作页需要大量表格录入，Element Plus 的表格和表单生态更贴合。

### 4.2 后端

推荐：`Spring Boot 3`

原因：

- 适合企业管理系统。
- 权限认证、事务控制、文件导出、日志审计实现稳定。
- 后续扩展审批流、回盘、发放状态也方便。

### 4.3 数据库

推荐：`MySQL 8`

用于保存：

- 用户和角色
- 人员库
- 工资计发单位库
- 工资批次
- 批次明细
- 导出记录
- 操作日志

### 4.4 权限认证

推荐：`JWT + RBAC`

规则：

- JWT 负责登录态。
- RBAC 负责菜单和接口权限。
- 普通用户按 `created_by` 隔离数据。
- 管理员不受 `created_by` 限制。

### 4.5 Excel 处理

推荐：`Apache POI`

原因：

- 当前模板同时包含 `.xlsx` 和 `.xls`。
- 建设银行模板是 `.xlsx`，中国银行模板是 `.xls`。
- Apache POI 同时支持两种格式，更适合按原模板写入并导出。

### 4.6 文件存储

第一版推荐本地文件存储。

规则：

- 导出的 Excel 文件保存到服务器本地目录。
- 数据库记录文件名、文件路径、导出人、导出时间、批次 ID、付款单位 ID。
- 后续如果部署到云服务器或多实例环境，可替换为对象存储。

## 5. 界面设计要求

整体风格：

- 清爽、专业、耐看，符合企业后台系统气质。
- 页面以工作效率为主，不做营销式大图和复杂装饰。
- 颜色保持克制，主色建议使用蓝色或青绿色，搭配中性色背景。
- 表格、表单、按钮间距要舒适，避免拥挤和廉价感。

工资批次制作页：

- 顶部展示批次信息和关键统计。
- 总人数、总发金额要突出显示。
- 明细表作为页面主体，支持长时间录入不疲劳。
- 常用操作按钮固定在明显位置，包括批量选人员、新增空行、批量设置摘要、保存、导出。
- 金额列要右对齐，便于核对。
- 错误信息要直接标在对应行或字段上，不只用弹窗提示。

交互要求：

- 批量选人弹窗要支持快速搜索和多选。
- 保存后提示清楚，但不要打断用户继续操作。
- 导出前校验要给出问题列表，并能定位到具体明细行。
- 管理员查看全部数据时，列表需要显示创建人，避免混淆。

## 6. 不在第一版范围内

- 银行提交接口
- 银行回盘导入
- 发放成功、失败状态跟踪
- 失败重发
- 审批流
- 项目、班组、客户单位、劳务队分组
- 复杂财务报表

这些能力可以作为第二阶段扩展。
